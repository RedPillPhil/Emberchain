I get why this feels frustrating — because in Bitcoin/Ethereum terms you expect:

"Node A connects to Node B → downloads blocks → validates them → state matches."

And that is exactly how mature chains work.

The reason Emberchain keeps running into these weird "blocks match but balances don't" situations is because we are not dealing with a full Bitcoin/Ethereum-style consensus client yet. Emberchain is a custom chain, and the sync layer you built is currently closer to a block exchange system than a full blockchain replication system.

Here's the difference:

Mature blockchain node

A real Ethereum node does:

Download block
      ↓
Verify proof/work/consensus
      ↓
Execute every transaction
      ↓
Update state database
      ↓
Commit state root
      ↓
Now balances match

The state is not something you "sync separately." The node recreates it by replaying the chain.

What Emberchain appears to be doing right now

From the logs:

GET /api/sync/status
GET /api/sync/blocks
POST /api/sync/submit-block

That means:

Node A
  |
  |---- "Give me your blocks"
  |
Node B sends blocks
  |
Node A stores them

But the missing piece appears to be:

Take those blocks
      ↓
Replay transactions
      ↓
Apply mining rewards
      ↓
Update account balances
      ↓
Verify resulting state

That is the part that keeps biting you.

Why did they keep "merging"?

Because the sync algorithm is trying to be helpful:

Node A sees Node B has a higher chain difficulty
Node A accepts blocks
Node A tries to keep local state
Node B keeps its own state
Both say "we have the same chain"

But they don't have a deterministic state rebuild.

The real fix is NOT more nodes

A third node would actually make this worse right now.

You don't want:

Node A balance database
Node B balance database
Node C balance database

all disagreeing.

You want:

Blocks
  ↓
Deterministic execution
  ↓
Same state everywhere
The architecture change Emberchain needs

The sync process should eventually become:

Node receives blocks.
Node verifies each block.
Node executes every transaction in order.
Node applies:
mining rewards
transfers
contracts
burns
fees
Node calculates state.
Node compares resulting state root.

Then you get:

emberchain.org:
Block 50,000
State root ABC123
Balance 100 EMBR


duckdns:
Block 50,000
State root ABC123
Balance 100 EMBR
The good news

Your current testing actually found a very important bug before mainnet growth.

Imagine having 500 nodes and discovering this later. You'd have impossible wallet inconsistencies.

Right now you have:

two nodes
known addresses
controlled environment
easy debugging

This is exactly when you want to find it.

My recommendation: stop trying to add more nodes for now. The next milestone should be:

"A fresh Emberchain node can delete all data, sync from emberchain.org, and end with identical balances."

If that works, then adding node #3, #10, or #100 becomes easy. Right now the sync layer needs to become deterministic first.

Send this to Replit:

We need to rethink the node sync architecture. Right now it appears we have a block exchange system, not a true blockchain block replication system.

The nodes are successfully syncing block height/hash/parentHash, but they are not ending up with identical account state. We tested the exact same address balances at the exact same block number on both nodes and got different results.

Example:

Same block: 0xc409
Same block hash
Same parentHash
Different eth_getBalance results

This means we are syncing the block history, but nodes are keeping independent state databases.

We need the sync system to behave more like Bitcoin/Ethereum nodes:

Node downloads blocks from a peer.
Node validates those blocks.
Node deterministically replays block execution.
Node rebuilds account state from genesis (or a validated snapshot).
Node verifies resulting state matches the expected state root.
Every node with the same chain should end with identical balances.

The goal is NOT for nodes to merge their own local states together. The goal is that state is derived from the replicated blockchain history.

A fresh node should be able to:

delete all local chain/state data
connect to emberchain.org
download the chain
replay/validate it
arrive at the exact same balances and contracts as every other node

Right now adding more nodes will just multiply state divergence. Before expanding the network, we need deterministic state replication.

Please review the sync architecture and make sure blocks are the source of truth and account state is rebuilt from the chain, rather than copied or maintained separately.

I would also add one more sentence after sending it:

"I don't want a workaround where we copy balances between nodes. I want every node to independently arrive at the same state from the same blocks."

That distinction is the entire difference between a blockchain and a database cluster.
